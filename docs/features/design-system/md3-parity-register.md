# MD3 parity register

## Mandate

The **entire** Bambu Studio GUI must conform to the vendored `ui-md3/design-system` kit
([`ui-md3/design-system/`](../../../ui-md3/design-system/), native token mirror
`src/slic3r/GUI/Widgets/MD3Tokens.hpp`). The target is **zero original (legacy Bambu) design
elements**: every surface, widget, dialog, overlay, icon, and font is either migrated to a kit
component/token or recorded as an explicit, deliberate deviation.

**Functional data colors are exempt.** Filament / AMS swatch colors, extrusion-role and G-code
feature palettes, tool-change bands, axis colors, printer product photography, brand logos,
instructional diagrams, and calibration-result imagery are *data*, not chrome. They are never
flagged and must be preserved through any restructure.

The color / token / typography layer is already substantially landed (see
[`md3-design-system.md`](md3-design-system.md)). The residual non-conformance this register tracks is
overwhelmingly three things:

1. **Legacy raster / SVG icons** where the kit specifies Material Symbols glyphs (the `MaterialIcon`
   helper exists but is adopted in only a handful of files);
2. **Legacy widget / dialog anatomy and geometry** — square radii, native OS window chrome, fixed
   pixel sizes, hand-rolled control shapes; and
3. **A few unmigrated panels and missing shared widgets** (no shared SearchField / Slider /
   SegmentedControl, no drawn Checkbox/Radio, no kit Dialog shell).

## How to use this register

- Each **surface** section below has one table of gaps. Columns, in order:
  **ID · Legacy element · Kit ref · Native anchor · Required change · Size · Risk · Status**.
- Every gap is **Status = open**. Close a row only when the change is implemented and verified
  in-app, in **light + dark**, and — where relevant — in the **Preview (purple)** and
  **Device (teal)** contextual schemes.
- **ID** is a stable handle; cite it in commits/PRs.
- **Size**: small / medium / large (implementation effort). **Risk**: low / medium / high
  (regression blast-radius).
- **Native anchor** paths are repository-relative (`src/slic3r/GUI/...`); line numbers were
  re-verified against the current branch tip by each auditor.
- **De-duplication.** Where two surfaces named the *same native anchor and the same change*, the gap
  is kept as one row in the **most specific** surface and removed from the other. Both the surviving
  row and the removed row carry a **Merge note**. Four such merges were applied (see the notes under
  the affected tables); the register holds **128** unique open gaps consolidated from **132** raw
  audit findings.
- The **Wave plan** sequences the open gaps into parallelizable implementation waves with disjoint
  file ownership per parallel group. The **Coverage** section quotes each auditor's coverage note
  **verbatim** so future sessions know exactly what was — and was not — audited.

## Wave plan

The 128 open gaps are sequenced below, ordered as instructed: **quick wins first, then
icon-replacement, then structural anatomy, then large/risky restructures last**. Within each wave the
**parallel groups own disjoint files**, so a group can proceed without blocking another. A handful of
very large files are contended by many gaps; those are declared as **single-owner serial tracks** and
appear in successive waves (the same owner returns to advance the file).

### Global prerequisites (build before the waves that depend on them)

- **MaterialIcon glyph enum** (`expand-materialicon-glyph-enum`, `Widgets/MaterialIcon.hpp`) — gates
  **every** wxDC raster→glyph swap (Wave 3, and the glyph half of many device/prepare rows).
- **ImGui font atlas** (`imgui-atlas-font-family`, `preview-imgui-mono-font`,
  `imgui-overlay-icon-glyphs`, all in `ImGuiWrapper.cpp`) — gates all ImGui overlay glyph/mono work
  (Preview timeline/statistics/Z-chip/options-chips, one-layer icon).
- **glyph→GL-texture bridge** (introduced by `viewport-main-toolbar-svg-sprites`,
  `GLToolbar.cpp`/`GLCanvas3D.cpp`) — gates all GL toolbar/gizmo icon work
  (`gizmo-rail-svg-icons`, and the icon half of the two GL toolbar reskins).
- **Shared kit widgets** — the drawn Checkbox/Radio, Switch, DropDown, field geometry
  (TextInput/ComboBox/SpinInput), SearchField, Slider, SegmentedControl, IconButton, and
  SectionHeader (all in `widgets-core`/`widgets-containment`) — gate their surface consumers
  (Prepare Process card, Device print-options, Settings Appearance, the dialog shells). These widgets
  are **large**, so by the size-ordering they land in Wave 6; every consuming medium/large card in
  Waves 5–7 that references a kit field/switch/segmented control carries a dependency edge on the
  corresponding Wave 6 widget and must be scheduled after it.
